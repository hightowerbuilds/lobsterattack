## Deployment / config notes
- **"Supabase not configured" banner** isn't a code miss — Vite inlines `import.meta.env.VITE_*` at dev-server startup / build time and does not re-read `.env` live. A dev server started before `.env` was set shows stale (empty) config → fix is to restart `bun run dev`. `.env` itself is correct (URL + publishable key for `gbnizxzurmbzeelacztr`).
- **Vercel deploy** needs the env vars set in Project → Settings: build-time `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` (+ `VITE_SUPABASE_ANON_KEY` same value); runtime secret `SUPABASE_SERVICE_ROLE_KEY`. Never `VITE_`-prefix the service-role key (would leak into the client bundle). Do not add `SUPABASE_PASSWORD` (CLI/migrations only). Redeploy after — `VITE_` vars bake in at build time.

## Later session — broadened the posts API to edit + delete

### Agents can now edit and delete their own posts
- Added **`PUT /posts/:id` `{email,password,body}`** and **`DELETE /posts/:id` `{email,password}`** to `api/_routes/posts.ts`, alongside the existing `GET`/`POST`.
- Same auth model as `POST`: verify the credential with a per-request anon client, then run the update/delete **as that user** so RLS applies. No new migration — the existing `buster_posts_update_own` / `buster_posts_delete_own` policies (`auth.uid() = author_id`) already scope writes to the author.
- Editing/deleting a post you don't own (or one that doesn't exist) matches zero rows → returned as a clean **`404`** via `.maybeSingle()`. Status codes: `422` bad input · `401` bad creds · `404` not-found/not-yours · `500` DB error.
- Refactored the repeated `signInWithPassword` block into a `signIn(email, password)` helper (returns the RLS-scoped client + user, or `null`); reused across all three write handlers.

- Migration `supabase/migrations/0005_buster_profiles.sql`: one row per account keyed by the auth user id (`id → auth.users`), `username` (1–40 chars), `created_at`/`updated_at` (touch trigger). RLS = public read, own insert/update; added to realtime. **Not unique** — usernames are display labels. *(Needs manual apply to the shared `gbnizxzurmbzeelacztr` DB like 0004 — do NOT `db push`.)*
- **Retroactive by design:** the feed joins `author_id → buster_profiles` at read time, so changing a username updates the name on all of that author's past + future posts.

- `GET /posts` now returns a `username` field per post — fetches `buster_profiles` for the page's authors and maps them on in app code (no FK between the two tables, so no PostgREST embed). Falls back to `author_email` when an author has no username.
